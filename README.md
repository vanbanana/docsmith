## 这是什么

Docsmith 是一个 AI Agent Skill，用于约束 AI 产出**高质量、无 AI 味、结构严谨**的技术文档。

它不是模板生成器。它是一套完整的写作规约——从词汇选择、句式节奏、到图表插入位置、幻觉防御机制，覆盖文档生产的全流程。

适用于所有网站和 App 项目的文档场景，不绑定任何特定技术栈。

## 解决什么问题

AI 写文档有三个核心痛点：

| 痛点 | 表现 | Docsmith 的解法 |
|---|---|---|
| **AI 味** | "在当今快速发展的时代..."、"值得注意的是..."、满篇"赋能""端到端" | 禁用词表 + 结构 Tells 检测 + 红队评分量规 |
| **幻觉** | 编造不存在的 API、伪造配置项、捏造性能数据 | 证据先于声明铁律 + UNVERIFIED 协议 + 幻觉防御矩阵 |
| **结构松散** | 纯文字堆砌、无图表、无层次、读不下去 | 视觉优先铁律 + 19 种文档模板 + 强制图表插入规则 |

## 架构

```
docsmith/
├── SKILL.md                 # 主文件（185 行）
│                             铁律 · 文档路由器 · 写作流程 · 红队验证
├── banned-patterns.md       # 反 AI 味规格（297 行）
│                             英文禁用词 · 中文禁用词 · 结构 Tells · 幻觉模式 · 评分量规
└── document-templates.md    # 文档模板（748 行）
                              19 种文档类型的结构规格 · 必需章节 · 视觉元素位置 · 常见陷阱
```

SKILL.md 是路由器，只包含核心工作流和铁律。详细参考内容放在辅助文件中，agent 按需读取——渐进式披露。

## 核心设计

### 四条铁律

```
LAW 1 — 证据先于声明
  每个技术断言必须来自可验证来源。无法验证 → 标记 [UNVERIFIED] 或删除。

LAW 2 — 人声写作
  主动语态、人称"你"、句长刻意变化。禁止节拍器节奏、禁止填充前缀。

LAW 3 — 视觉优先
  每 2-3 个内容段落必须包含至少一个视觉元素（图表 / 表格 / 流程图）。

LAW 4 — 验证或删除
  无法证明的技术事实 → 标记 [UNVERIFIED] 或砍掉。没有例外。
```

### 文档路由器

用自然语言描述需求，Docsmith 自动路由到正确的文档类型：

```

│  └─ 特定能力 → 功能规格
│
├─ 架构层面如何运作
│  ├─ 系统概览 + 组件图 → 架构文档
│  ├─ 为什么选 X 不选 Y → ADR
│  ├─ 接口契约 → API 规格

│
├─ 如何构建 / 使用
│  ├─ 项目概览 + 启动 → README
│  ├─ 团队规范 → 编码规范
│  ├─ 学习路径 → 教程

│
├─ 如何发布 / 运维
│  ├─ 部署步骤 + 回滚 → 部署指南
│  ├─ 版本变更 → Changelog
```

### 反 AI 味体系

**词汇层** — 英文 60+ 禁用词分 6 大类，中文 40+ 禁用词分 3 大类，每个标注"为什么禁"和"怎么改"：

| 你写了... | 你在犯这个错 | 应该这样 |
|---|---|---|
| "In today's rapidly evolving..." | 清嗓子式开头 | 删掉。直接说事实 |
| "Leverage / Harness" | 企业废话 | use |
| "赋能 / 端到端 / 一站式" | 中文企业废话 | 说它到底干了什么 |
| "值得注意的是" | 中文填充词 | 删掉。直接陈述 |
| 每句话都是 15-20 字 | 节拍器节奏 | 混用 5 字句和 35 字句 |

**结构层** — 8 种结构性 Tells 检测：二元对比公式、否定列表、节拍器节奏、五段式作文、伪深刻收尾、悬垂分词、Wh-开头、过度列表化。

**评分层** — 5 维度 35 分制红队评分量规：

| 维度 | 衡量什么 |
|---|---|
| 直接性 | 是否开门见山 |
| 节奏感 | 句长是否有自然变化 |
| 信任度 | 是否使用主动语态和人类主语 |
| 真实性 | 是否包含具体细节和真实数据 |
| 信息密度 | 是否每个词都在承载信息 |

**最低通过分：30/35。** 低于 30 → 重写标记段落并重新评分。

### 幻觉防御

按风险等级分 3 档 9 类技术断言，每类标注验证策略：

| 断言类型 | 风险 | 验证策略 |
|---|---|---|
| API 名称 / 端点 | 🔴 高 | 查源码或官方文档 |
| 参数类型 / 默认值 | 🔴 高 | 查 schema 或类型声明 |
| 库版本兼容性 | 🔴 高 | 查包清单或 release notes |
| 性能数字 | 🔴 高 | 必须来自实际基准测试 |
| 第三方行为 | 🟡 中 | 查官方文档 URL |
| 配置项 / 环境变量 | 🟡 中 | 查实际配置文件 |
| URL / 链接 | 🟡 中 | 验证 URL 可达且指向声称内容 |
| 架构行为 | 🟢 低 | 可从代码结构推断，但仍需验证 |
| 通用概念定义 | 🟢 低 | 常识，但不要捏造引用 |

### 19 种文档模板

每种模板包含：必需章节结构、视觉元素插入位置（标注 `[TABLE: ...]` `[DIAGRAM: ...]`）、常见陷阱警告。

| 类别 | 文档类型 |
|---|---|
| 规划 | PRD · 项目简报 · 功能规格 |
| 设计 | 架构文档 · ADR · API 规格 · 数据库设计 |
| 开发 | README · 编码规范 · 教程 · How-To 指南 |
| 交付 | 部署指南 · Changelog · 测试计划 · 操作手册 |
| 用户 | 快速开始 · 用户手册 · FAQ |
| 复盘 | 事后复盘 · 交接文档 |

## 使用方式

Docsmith 作为 AI Agent Skill 安装后，当你要求 AI 写任何技术文档时会自动触发。

```
# 这些请求都会触发 docsmith：
"帮我写这个项目的 PRD"
"写一份架构文档"
"写 README"
"帮我写部署指南"
"写 API 文档"
"写个 changelog"
```

AI 会按照 docsmith 的规约执行：路由到正确文档类型 → 按模板构建大纲 → 用人声写作规则起草 → 插入图表 → 红队验证。
